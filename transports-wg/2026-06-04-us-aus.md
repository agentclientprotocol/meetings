## Agenda

- Introductions and Project Context
  - Alex Hancock: Software engineer at Block, using ACP as default interface for Goose
  - Clifford Ressel: CTO at Nori, uses ACP for remote-hosted agents and TUI clients, durability is central concern
  - Kaiser Dandangi: Software engineer at Atlassian, supporting product teams building agents
  - Federico Ciner: Atlassian, building new core agent runtime platform, developed TypeScript implementation of the RFD
  - Josh Voigts: Works in cybersecurity

- Protocol Overview and V1/V2 Strategy
  - Consensus that spec is generally strong but durability and connection reliability need refinement
  - V2 will include primitives to address durability/reliability issues
  - Acceptable to ship V1 now; developers implement what makes sense for current needs
  - Full durability and reliability solidified in V2 and potential future RFDs

- Implementation Challenges and Architectural Feedback
  - Atlassian's microservice architecture lacked native sticky session support
  - Deviated by storing session and connection IDs in distributed Redis streams database
  - Introducing message ID / "last replay ID" into session context would improve retry capabilities
  - V1 sufficient for production use; teams handle connection timeouts and retry logic within their frameworks

- Session and Connection Lifecycles
  - Connection establishes specific capabilities
  - Session is hydrated through a "load" call, persists even if client disconnects and reconnects
  - Multiple connections interfacing with single session is a manageable server-side scaling challenge

- Stateless Architecture Considerations
  - MCP working group exploring stateless approach to simplify operations and scaling
  - ACP relies on bidirectional message passing that may be difficult in strictly stateless environment
  - Needs further evaluation for V2

- Resilience Through Proxies and Ping-Pong Mechanisms
  - Proxy pattern advocated for durability: transparent websocket reconnection and consistent packet handling
  - Transport-level pings prevent intermediate network devices from timing out connections
  - Application-level pings ensure underlying process is responsive (detect half-open connections, resource-constrained environments)

- Documentation and Next Steps
  - Table clarifying V1 vs V2 functionality and resiliency features SDK implementations should support
  - Spec to move to "preview" stage once clarifications documented
  - Will remain in preview while production implementations mature

## Decisions

- V1 protocol spec approved for production use; durability/reliability enhancements deferred to V2
- Protocol status set to "preview" to allow SDK iteration and production testing before stable release
- Stateless architecture for V2 needs further discussion to determine impact on server-to-client messaging
